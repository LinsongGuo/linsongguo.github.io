Aspen-Go limitation #1: infrequent packet polling

Aspen-KB
Kernel-bypass network stack → poll on every preemption.
Aspen-Go
Unmodified Go: expensive network stack (no kernel bypass) → poll only when scheduler runqueue is empty.
Aspen-Go: offload polling to a timer core (polls every 100 µs).
Aspen-Go is weaker at preventing head-of-line blocking.

Example: existing interfaces amplify lock contention

What info could schedulers leverage?

Urgency

How soon must the wakee run?

Placement

Should wakee and waker run in parallel?

Thread relationships

KEY INSIGHT
1. Libraries already know or can infer this.
2. They can convey it to the scheduler without modifying applications.

Proposed interface: thread_wakeup() with intent

void thread_wakeup(thread_t *wakee,
                   enum wakee_urgency,
                   enum placement);

wakee_urgency

HIGH
cannot tolerate any notification/migration delay
MEDIUM
short migration delay OK
NONE
no urgency

placement

SEPARATE
run waker and wakee on different cores in parallel
ANY
no placement preference

Group interface: thread_wakeup_group(thread_t[], PARALLEL or ANY)

Example: thread_wakeup(waiter, HIGH, SEPARATE)

the library calls the new interface with intent HIGH × SEPARATE. HIGH: waiter 1 must run on the current core now — but waker still running. waker forced to yield. waiter 1 takes the core. fast lock handoff, contention relieved. but wait — how does the scheduler handle the waker? the scheduler picks a remote core. SEPARATE: waker takes the remote core by preemption. waker lands on remote; both run in parallel.

Performance on lock-contention workloads

  • Memcached workload: GETs only.
  • Each GET acquires a lock on its key; keys with the same hash share the lock.
  • Key accesses follow a skewed Zipfian distribution (α=0.99).
    Top 10% of keys receive ~80% of GETs.

Apps share most of their semantics

Methodology: Used LLMs to analyze 20 apps and identify semantics for each app.
Conclusions: 6 semantics are commonly used; 21 total across all 20.
The 6 common semantics Description Used in (apps)
Thread role Is this thread user-facing or background? 20 / 20
Blocking Is this thread blocking others, and how many are waiting? 20 / 20
Internal queue depth How much work is waiting for this thread to handle? 19 / 20
Request type Which request is this thread serving? 18 / 20
Performance stall warning Will the app stall if this thread doesn’t run soon? 16 / 20
Co-scheduling group Should this thread run simultaneously with others? 9 / 20
Apps surveyed: Memcached, Redis, Elasticsearch, Kafka, Storm, Spark, RocksDB, Nginx, CoreDNS, Firecracker, ZeroMQ, vLLM, MySQL, PostgreSQL, Cassandra, CockroachDB, FFmpeg, etcd (+ 2 more).

Open challenges

Three challenges to make semantic scheduling deployable.
1

Runtime channel.

How can apps deliver live semantics into the kernel scheduler efficiently?
2

Coexistence.

How should a semantic-aware policy co-exist with threads that have no LLM analysis?
3

Verification.

Can we check the LLM-generated policies for no-starvation and bounded latency before deployment?
