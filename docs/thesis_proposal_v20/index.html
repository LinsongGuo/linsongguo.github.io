Existing schedulers preempt application cores periodically. For example, the Linux scheduler sends preemption every few milliseconds. However, when preemption is at microsecond scale, periodic preemption becomes expensive. Even though user interrupts have lower overhead, they're still not free.

Instead, our policy is to

[CLICK 1 — policy reveal]

preempt only when necessary. If there are no urgent tasks waiting, there's no need to preempt the currently running task.

Specifically, the timer core

[CLICK 2 — timer checks queues]

periodically checks the network stack and scheduler runqueue. If the timer core finds any task waiting,

[CLICK 3 — user interrupt fires]

it issues a user interrupt to preempt the currently running task. Otherwise, it skips the preemption.

With this approach, we can

[CLICK 4 — 70% savings number]

avoid over 70% of preemption events. This greatly reduces preemption cost.

Extended User Interrupts (xUI):
Fast and Flexible Notification without Polling

Berk Aydogmus, Linsong Guo, Danial Zuberi, Tal Garfinkel,
Dean Tullsen, Amy Ousterhout, Kazem Taram
ASPLOS 2025 ★ Best Paper Award ★ IEEE MICRO Top Picks 2026

In this work, we design hardware extensions to user interrupts that further reduce preemption overhead. This is published at ASPLOS 2025.

Limitations of user interrupts

Even though user interrupts already help, they still have a high overhead.

First, when app cores are preempted, they spend around 700 cycles handling the interrupt. That's a lot of CPU cycles taken away from the actual workloads. When preemption happens every few microseconds, the cost adds up quickly.

Second, Aspen uses a dedicated timer core to fire preemptions.

[CLICK — timer-core panel + chart appear]

The timer core spends 400 cycles sending one user interrupt. This is a high cost, especially when preemption is very frequent — like every 5 microseconds. The figure shows how many app cores one timer core can preempt. The x-axis is the preemption quantum — how often we preempt — and the y-axis is how many cores it can keep up with. At ten microseconds it handles all 24, but at one microsecond it drops to one. This indicates the timer core design cannot scale to many cores.

xUI extension #1: avoid pipeline flushing

 

Now I'll talk about the optimizations of xUI. We have talked about the overhead on app cores. Where does that overhead come from? When a user interrupt arrives, the CPU flushes the whole pipeline. We try to avoid that.

Let me start with a quick review of a processor pipeline. The front-end fetches and decodes instructions, and they enter the buffer and start to run. The buffer can hold hundreds of them in flight.

[CLICK 1–2 — buffer advances]

Each cycle, one instruction commits and retires, and a new one enters the buffer.

[CLICK 3 — ⚡ interrupt arrives]

Now an interrupt arrives. Some instructions are still in flight.

[CLICK 4 — FLUSH]

With today's user interrupts, the CPU flushes them all.

[CLICK 5 — handler fills]

Then the handler runs. But the program work is lost.

[CLICK 6 — xUI pipeline appears]

Now let's look at our design with the same starting state.

[CLICK 7 — ⚡ xUI interrupt]

At some point, an interrupt arrives.

[CLICK 8 — handler injected at tail]

Instead of flushing, we simply inject the handler at the tail. No flush. Program instructions still commit.

[CLICK 9–10 — xUI cycles advance]

Handler and program instructions run together. No program instructions are lost.

xUI extension #2: hardware timer

We also talked about overhead on the timer core. Aspen has a dedicated timer core that busy-spins and preempts app cores.

[CLICK 1 — arrows + handler pills]

The timer core sends interrupts to app cores and invokes the interrupt handler.

[CLICK 2 — ~400 cycles per send]

It takes about 400 cycles for each send.

[CLICK 3 — punch line + xUI panel appears]

Because of the overhead, one dedicated timer core can only preempt a limited number of app cores. We fix this with a hardware timer inside each core.

[CLICK 4 — timer pills appear]

Each core has its own timer.

[CLICK 5 — arrows + handler pills]

When a timer fires, it triggers the interrupt handler locally. There's no cross-core traffic.

[CLICK 6 — punch line]

So we don't need a dedicated timer core anymore. Fine-grained preemption scales with the number of cores.

On the left is the mutex. It is held by the thread on the current core. This thread will release the lock.

[CLICK 1 — unlock]

The waker thread calls mutex_unlock. The lock is released. Now the runtime needs to wake one waiter. The waiter queue has four threads blocked on this mutex. Waiter 1 is at the head, so the runtime wakes up waiter 1.

[CLICK 2 — thread_ready]

Then the runtime calls the interface thread_ready on waiter 1. But thread_ready only says waiter 1 is ready. It doesn't convey any urgency information.

[CLICK 3 — contention amplified]

This causes a cascading effect. The other waiters cannot acquire the lock until waiter 1 runs and releases the lock. So all the waiters are delayed by waiter 1's queue wait.

From the previous example, we see urgency is missing. But urgency is not the only dimension. The interface could convey more.

Urgency says how soon the wakee must run. For example, the waiter from lock contention must run immediately, while background work has no urgency.

[CLICK 1 — Placement card]

Placement says whether the wakee should run in parallel with the waker. If both are urgent, separate cores let them run in parallel.

[CLICK 2 — Thread relationships card]

Thread relationships say how co-woken threads relate. For example, a reader-writer lock or barrier wakes many threads at once. They could benefit if we run them in parallel.

So all three dimensions could help the scheduler make better decisions.

[CLICK 3 — KEY INSIGHT]

Our insight is that the library has all three already. For example, the library can detect lock contention by tracking how long a waiter has been waiting for one lock. So the library can know this without changing application code.

Proposed interface: thread_wakeup() with intent

Group interface: thread_wakeup_group(thread_t[], PARALLEL or ANY)

Based on this insight, we propose a new interface called thread_wakeup. It takes the wakee thread, and two dimensions of information.

For urgency, we have three levels.

The difference between HIGH and MEDIUM is migration cost. Migrating a thread to another core has overhead. So if the wakee is very urgent and cannot tolerate this overhead, we should use HIGH urgency. One example is the lock contention case. If we don't wake the waiter immediately, there will be a cascading effect for all waiters of the lock. Otherwise, MEDIUM is fine.

For placement, SEPARATE means run the waker and wakee on different cores in parallel. ANY means no placement preference.

[CLICK 1 — group interface]

We also have a group interface called thread_wakeup_group. It takes a list of threads and a placement option. This is for cases like barriers or reader-writer locks, which would wake up many threads at once. They could benefit from running in parallel.

Example: thread_wakeup(waiter, HIGH, SEPARATE)

the library calls the new interface with intent HIGH × SEPARATE. HIGH: waiter 1 must run on the current core now — but waker still running. waker forced to yield. waiter 1 takes the core. fast lock handoff, contention relieved. but wait — how does the scheduler handle the waker? the scheduler picks a remote core. SEPARATE: waker takes the remote core by preemption. waker lands on remote; both run in parallel.

Now let me show how the new interface addresses the lock contention. The library uses thread_wakeup with HIGH and SEPARATE.

[CLICK 1 — HIGH: slot reserved]

First, the library conveys urgency HIGH to the scheduler. HIGH means waiter 1 must run on this core immediately. So the scheduler holds the CPU for waiter 1. But the waker thread is still running there.

[CLICK 2 — waker yields]

So the waker thread is forced to yield. This frees the CPU for waiter 1.

[CLICK 3 — waiter 1 takes the core]

Then waiter 1 takes the CPU. Now it is holding the lock and running.

[CLICK 4 — contention relieved]

The lock handoff is very quick. And the other waiters will also benefit. When waiter 1 releases the lock, it can wake them up using the same fast handoff.

[CLICK 5 — but wait]

But one problem is, where did the waker go?

[CLICK 6 — remote core appears]

The scheduler picks a remote core. This is where SEPARATE comes in: the library conveys that the wakee and waker can run on separate cores in parallel. But the remote core is not free. It is already running another thread.

[CLICK 7 — UIPI preempts]

So the scheduler sends a user interrupt to preempt the remote thread.

[CLICK 8 — waker lands on remote]

The waker takes the CPU. Now waiter 1 on the current core and the waker on the remote core run in parallel. So SEPARATE is enforced.

Making threads non-anonymous

To answer this question, we propose to make threads non-anonymous.

[CLICK 1 — LEFT struct]

Today, the thread struct only has system-level fields, like tid, state, and runtime.

[CLICK 2 — anonymous caption]

So the scheduler does not know what each thread is doing. For example, is this thread user-facing or background?

[CLICK 3 — RIGHT struct skeleton]

Our solution is to add a semantic context to the thread struct. This semantic context is different per app.

[CLICK 4 — thread_role tags]

The first field is thread_role. It tells whether this thread is user-facing or background. So the scheduler can prioritize user-facing threads.

[CLICK 5 — more fields]

We also have other fields. For example, request_type tells what request this thread is doing, like whether it is short or long.

Another field is write_stall_warning. This is in RocksDB. RocksDB buffers writes in memory. A background thread called flush writes the data to disk. Even though it runs in the background, the flush thread still needs to run in time. Otherwise, the buffer will fill up, and new write requests will stall. So write_stall_warning tells whether writes will stall in RocksDB.

[CLICK 6 — non-anonymous label]

Now threads are non-anonymous. The scheduler knows what each thread is doing.

Now let me introduce the idea of semantic scheduling. Consider the same RocksDB scenario from earlier. The flush thread must run in time to avoid write stalls. How can the app give a warning to the scheduler?

[CLICK 1 — LEFT TODAY]

Today, the app uses interfaces like sched_setscheduler and sched_setaffinity. The app has to pick a priority and a core. But this is not easy. There is a tradeoff here. Flush must run in time, but user-facing threads and other apps also matter. If the given priority is too high, flush blocks user-facing threads and other apps. If too low, flush does not run in time, and writes stall. The app has only a narrow view, so it cannot navigate this tradeoff. The given priority and core are just guesses. The scheduler just follows whatever the app picked.

[CLICK 2 — RIGHT SEMANTIC SCHEDULING]

With semantic scheduling, the app describes what the thread is. It sets thread_role to BACKGROUND and request_type to FLUSH. Note that write_stall_warning is a runtime signal. Its value depends on the current buffer state, so it changes as the system runs. When the warning flows into the scheduler, the scheduler decides priority and core from a global view. With a global view, the scheduler can navigate the tradeoff. For example, it can give flush just enough priority to avoid the stall. User-facing threads and other apps won't be blocked too much.

So the key idea is: the app informs, and the scheduler decides.

How do we design a scheduler? The scheduler has two parts: mechanism and policy.

[CLICK 1 — mechanism appears]

On the left is the mechanism. The mechanism is the data structures and primitives in the scheduler, like the runqueue, preemption, and migration. They are implemented by us.

[CLICK 2 — mechanism why-line]

The mechanism does not rely on app semantics. So it stays fixed across all apps.

[CLICK 3 — policy appears]

On the right is the policy. The policy has some hook functions. The signatures are fixed by us, but the bodies are LLM-generated. The separation of mechanism and policy brings two benefits. When a new app comes, only the policy needs to be reloaded. The mechanism doesn't change. Also, since the hook functions have fixed signatures, they are easy to understand and verify. These functions take app semantics as input and return scheduling decisions. For example, priority returns priority. placement picks a CPU. time_slice returns a duration. should_preempt decides whether a new thread bumps the running one.

[CLICK 4 — policy why-line]

The policy relies on app semantics. So it's reloaded when apps change.

[CLICK 5 — call policy hooks arrow]

The mechanism calls the policy hooks to get decisions.

[CLICK 6 — runqueue example]

Take the runqueue as an example. When the app updates semantics, the scheduler calls the priority hook. Then the scheduler uses the new priority to reorder threads.
